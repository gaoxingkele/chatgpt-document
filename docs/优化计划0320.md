# 报告质量优化计划 2026-03-20

## 背景

基于 kateer 语料（492 篇反毒品新闻）的完整管线测试，v1 评分 38/100，经 Step3b 评估驱动改写后 v3 提升至 62/100。主要短板：文本质量 58、章节递进 52、重点突出 59。本计划从 Skill 体系、文档结构、多模态内容、质量迭代四个层面提出改造方案。

## 当前管线架构

```
Step0b(预处理) → Step2(v1) → Step3b(评估) → Step5(v3) → Step6(引用) → Step7(风格化) → Step9(润色)
```

新管线已跳过旧 Step3(泛化专家)+Step4(膨胀重写)，用 Step3b 领域自适应评估替代。

## 优化一：Skill 体系重构

### 现状问题

当前 Skill.md 是泛化的"学术写作规范"，对所有报告类型用同一套——等于没有 Skill。policy 目录只有 Skill.md + summary.md 两个文件，缺乏结构模板和质量标准。

### 改造方案

每个 policy 目录扩展为：

```
policy_political_commentary/
├── Skill.md              ← 写作规范（已有，需细化为领域专属）
├── summary.md            ← 风格摘要（已有）
├── structure_template.md ← 新增：章节结构模板
├── examples/             ← 新增：优秀段落示例
│   ├── good_analysis.md  ← "审慎平衡"的范文片段
│   └── bad_patterns.md   ← 要避免的 AI 味写法
└── evaluation_rubric.md  ← 新增：评分标准（喂给 Step3b）
```

### 各类型 Skill 差异化

| 报告类型 | Skill 核心内容 | 结构模板关键元素 |
|---|---|---|
| 政治评论 | 利益相关方分析框架、政策因果链模板、争议标注规范、审慎表述库 | 事件→分析→预测 |
| 学术研究 | 文献综述结构、研究方法论段落、理论框架模板、引用规范 | IMRAD 变体 |
| 商业分析 | 商业画布模板、财务指标段落、竞争格局分析框架 | Porter/SWOT 嵌入 |
| 可行性研究 | 风险矩阵、技术路线图、成本收益模板、决策树 | 技术→经济→法规→风险→建议 |

### `structure_template.md` 示例（政治评论）

```markdown
# 推荐结构

1. 背景与问题界定（WHY：为什么关注此议题）
2. 关键事件时间线（WHAT：发生了什么）
3. 利益相关方分析（WHO：各方立场与动机）
4. 政策因果链分析（HOW：政策如何传导影响）
5. 争议与替代解释（BUT：反面证据与不确定性）
6. 前景评估与政策含义（SO WHAT：对决策者意味什么）

# 每章必须包含

- 至少一个具体案例或数据点
- 至少一个对立观点或不确定性标注
- 利益相关方的明确立场引述
- Mermaid 图表（时间线/关系图/流程图，视内容选择）
```

### `evaluation_rubric.md` 示例

```markdown
# 评分标准（供 Step3b 使用）

## 90+ 优秀
- 每个论点有多源交叉验证
- 对立观点充分展开而非一笔带过
- 时间线精确到日，利益相关方立场有直接引述

## 70-89 良好
- 关键事实有来源支撑
- 主要争议有提及
- 章节间有明确逻辑递进

## < 60 不合格
- 出现无来源的事实断言
- 缺乏对立视角
- 章节间逻辑跳跃
```

### 实施要点

- Step2 大纲生成时注入 `structure_template.md`（已有读取机制，`report_type_profiles.py` 的 `大纲约束` 段落）
- Step3b 评估时注入 `evaluation_rubric.md` 作为评分标准
- Step5/Step7 改写时注入 `examples/good_analysis.md` 作为风格参考
- Step7 已读取 Skill.md + summary.md，无需改动读取逻辑

## 优化二：文档结构差异化

### 现状问题

所有报告类型走同一个 Step2 大纲生成流程，结构趋同。政治评论和学术研究输出的章节结构几乎一样。

### 改造方案

在 `report_types/*.md` profile 中丰富 `## 大纲约束` 段落，Step2 的 `_api_build_outline` 已有注入机制。

### 各类型结构模板

**政治评论**：事件驱动 + 多视角分析
```
1. 背景与威胁评估
2. 关键事件与时间线
3. 利益相关方博弈分析
4. 政策因果链与传导机制
5. 争议焦点与替代解释
6. 前景研判与政策建议
```

**学术研究**：问题驱动 + 方法论
```
1. 引言与研究问题
2. 文献综述与理论框架
3. 研究方法与数据来源
4. 研究发现
5. 讨论与理论贡献
6. 结论、局限与未来方向
```

**商业分析**：市场驱动 + 决策导向
```
1. 执行摘要
2. 市场环境分析（PEST/五力）
3. 竞争格局与标杆对比
4. 商业模式与价值链
5. 财务分析与预测
6. 战略建议与路线图
```

**可行性研究**：决策驱动 + 风险评估
```
1. 项目概述与目标
2. 技术可行性
3. 经济可行性（成本收益分析）
4. 法规合规性
5. 风险评估与缓解
6. 结论与决策建议
```

### 实施要点

- 修改各 `report_types/*.md`，补充 `## 大纲约束` 段落
- Step2 的 `_api_build_outline` 已读取此段落并注入 prompt，无需改代码
- 效果：章节递进分数预期 +15-20

## 优化三：多模态内容

### 3.1 Mermaid 图表（零成本，P0）

在 Step5/Step7 的改写 prompt 中要求 LLM 在适当位置生成 Mermaid 代码块。

**适用图表类型**：

| 报告类型 | 推荐图表 | Mermaid 语法 |
|---|---|---|
| 政治评论 | 事件时间线 | `timeline` |
| 政治评论 | 利益相关方关系 | `graph TD` |
| 政治评论 | 政策因果链 | `flowchart LR` |
| 学术研究 | 研究框架 | `graph TD` |
| 商业分析 | 商业模式画布 | `graph TD` |
| 商业分析 | 市场份额 | `pie` |
| 可行性研究 | 决策树 | `flowchart TD` |
| 可行性研究 | 风险矩阵 | `quadrantChart` |

**示例 prompt 注入**：
```
在适当位置插入 Mermaid 图表（用 ```mermaid 代码块）：
- 涉及时间线的内容：用 timeline 图
- 涉及多方关系的内容：用 graph 关系图
- 涉及因果/流程的内容：用 flowchart 流程图
每章至少一个图表，使报告更直观。
```

**DOCX 渲染**：`docx_utils.py` 需扩展 Mermaid → PNG 转换（可用 mermaid-cli 或在线 API）。HTML 导出原生支持。

### 3.2 Markdown 表格增强（零成本，P0）

在 prompt 中明确要求：
- 数据对比必须用表格，不用列表
- 利益相关方立场用对照表（方 | 立场 | 动机 | 行动）
- 时间线事件用结构化表格（日期 | 事件 | 来源 | 影响）

### 3.3 数据可视化（P2）

**利用 Step0b Mode B 的知识图谱数据**：
- 用 `networkx` + `matplotlib` 生成实体关系图 PNG
- 用 `matplotlib` 生成词频/实体统计柱图
- 嵌入报告的 Corpus Package（`assets/` 目录）

**HTML 报告增强**：
- 嵌入交互式 ECharts 图表
- 可折叠的引用详情
- 时间线滑动浏览

### 3.4 实施路径

```
Phase 1（零成本）: Mermaid prompt 注入 + 表格规范 → Step5/Step7 prompt 修改
Phase 2（轻量级）: Mermaid → PNG 渲染 → docx_utils.py 扩展
Phase 3（进阶）: matplotlib 统计图 + HTML 交互报告
```

## 优化四：质量驱动迭代闭环

### 现状问题

Step3b 评估一次，Step5 改写一次，不管分数是否达标就输出。62 分的报告未达到"良好"（70+）标准。

### 改造方案

```
v1 → Step3b(评估)
        ↓
    分数 ≥ 目标? ──是──→ 直接进入 Step6
        ↓ 否
    Step5(改写, 注入评估意见) → Step3b(再评估)
        ↓
    分数 ≥ 目标? ──是──→ 进入 Step6
        ↓ 否
    Step5(只改写最低分章节) → 输出（最多 2 轮）
```

### 迭代策略

| 轮次 | 改写范围 | 评估方式 | 退出条件 |
|---|---|---|---|
| 第 1 轮 | 全部章节 | 五维度全评 | 总分 ≥ 70 或各维度均 ≥ 60 |
| 第 2 轮 | 仅评分 < 60 的章节 | 只评低分维度 | 任意条件达标或到上限 |

### 加权策略

第二轮 Step5 prompt 中，对低分维度加权：
```
本章在「章节递进」维度评分仅 35/100，请特别注意：
- 与上一章的逻辑衔接
- 从具体到抽象的递进层次
- 过渡段落的自然性
```

### 成本控制

- 第二轮只改写低分章节（如 6 章中只有 2 章 < 60），节省 66% token
- 第二轮评估只跑低分维度（不做 SOTA 搜索），节省 Perplexity 调用
- 总额外成本：约 +30% token（1 轮改写 + 1 轮评估）

## 优化五：语言策略规范

### 现状问题

英文语料 → 中文报告，但大量英文原文残留（人名、机构名、引述），中英混杂影响可读性。

### 规范方案

在 Skill.md 或 Step5 prompt 中注入：

```markdown
# 语言规范

## 专有名词
- 首次出现：中文译名（英文原名），如"贾利斯科新世代卡特尔（CJNG）"
- 后续引用：使用缩写或中文简称，如"CJNG"
- 人名：中文音译 + 英文原名，如"奥塞格拉·塞万提斯（Nemesio Oseguera Cervantes）"

## 直接引述
- 保留英文原文，用引号标注
- 提供中文翻译，如：美方声明称"non-state armed groups"（非国家武装团体）

## 数据与来源
- 统一格式：据[来源名称]报道/数据
- 日期统一为 YYYY年M月D日 格式
- 货币统一标注，如"320亿美元"

## 避免
- 整段英文不翻译直接引用
- 中英文混用的半句话（如"进行了counter cartel行动"）
- ISO 时间戳、裸 URL 出现在正文中
```

## 优化六：Step3b 评估精度提升

### 现状问题

Step3b 用单次 LLM 调用做五维度评估，可能不够精准——一个 prompt 同时评 5 个维度容易互相干扰。

### 改造方案

**分维度评估**：每个维度单独调用一次 LLM（5 次并行），每次只聚焦一个维度，评估更深入。

**注入领域评分标准**：读取 `evaluation_rubric.md`，替代通用标准。

**章节级评估**：当前是全文一次评估，改为逐章评估 + 全文总评。每章输出各维度分数，Step5 可按章节精准注入修改意见。

### 预期效果

- 评估精度提升：各维度分数方差更小，修改意见更具体
- 改写效果提升：章节级意见比全文级更精准
- 额外成本：5 次并行调用 vs 1 次，token 量相当（输入重复但输出更专注）
