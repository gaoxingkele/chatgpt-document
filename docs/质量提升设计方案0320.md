# 报告质量提升设计方案 2026-03-20

## 设计决策

- **搜索引擎**：不用 Tavily，保持 Perplexity 直连为默认；复杂研究场景集成 GPT-Researcher（改造其搜索源为 Perplexity/Grok/Gemini，推理引擎接入 llm_client 路由）
- **评估框架**：Rubric-Driven + REVISEVAL 式自动修正
- **结构模板**：从 RAND/Brookings/Stanford 等权威机构获取各类型模板
- **多模态**：Mermaid 图表 + 表格规范
- **Prompt 工程**：结构化 prompt 持续优化

---

## 一、信息获取层改造

### 1.1 双模搜索策略

| 场景 | 引擎 | 触发条件 | 示例 |
|---|---|---|---|
| 事实匹配 | Perplexity sonar-pro 直连 | Step6 引用、Step3b SOTA、Step9 补注 | "El Mencho 何时被击毙" |
| 深度研究 | GPT-Researcher | 用户显式 `--deep-research` 或评估发现重大遗漏 | "拉美反毒政策的学术争议全景" |

### 1.2 GPT-Researcher 集成方案

**改造目标**：将 GPT-Researcher 的搜索源和推理引擎接入现有 llm_client 路由，不引入 Tavily。

```python
# 改造后的 GPT-Researcher 配置
class ResearchConfig:
    # 搜索源：优先 Perplexity，可切换 Grok/Gemini
    SEARCH_PROVIDER = "perplexity"  # perplexity | grok | gemini

    # 推理引擎：复用 llm_client.chat()
    REASONING_PROVIDER = None  # None=跟随 LLM_PROVIDER
    REASONING_MODEL = None     # None=跟随 GROK_REASONING_MODEL

    # 研究参数
    MAX_SEARCH_ITERATIONS = 5   # 最大搜索轮次
    MAX_SOURCES_PER_QUERY = 10  # 每次搜索最大来源数
    REPORT_FORMAT = "markdown"  # 输出格式
```

**集成点**：
- `src/research/` 新包，封装 GPT-Researcher 核心逻辑
- 搜索适配器：`PerplexitySearchAdapter`、`GrokSearchAdapter`、`GeminiSearchAdapter`
- 推理引擎：直接调用 `llm_client.chat(reasoning=True)`
- Step9 中：`--deep-research` 时调用 GPT-Researcher，否则 Perplexity 直连

**参考**：
- [GPT-Researcher](https://github.com/assafelovic/gpt-researcher) — 核心 agent 逻辑
- [LangChain Open Deep Research](https://github.com/langchain-ai/open_deep_research) — 替代参考

### 1.3 搜索源适配器设计

```python
class SearchAdapter:
    """搜索源统一接口"""
    def search(self, query: str, max_results: int = 10) -> list[SearchResult]:
        ...

class PerplexitySearchAdapter(SearchAdapter):
    """复用现有 perplexity_chat_with_citations"""
    # 优势：自带引用、成本低、速度快
    # 劣势：单次搜索深度有限

class GrokSearchAdapter(SearchAdapter):
    """Grok 实时搜索（X/Twitter 数据优势）"""
    # 优势：实时性强、社交媒体数据
    # 劣势：引用格式不统一

class GeminiSearchAdapter(SearchAdapter):
    """Gemini + Google Search grounding"""
    # 优势：学术文献覆盖好
    # 劣势：需要额外 grounding 配置
```

## 二、结构模板层

### 2.1 权威模板来源

| 报告类型 | 模板来源 | 核心结构 |
|---|---|---|
| 政治评论 | [RAND 政策分析框架](https://www.rand.org/pubs/papers/P4156.html) + [Brookings 简报](https://www.brookings.edu/tags/brookings-policy-brief-series/) | 问题界定→利益相关方→政策选项→权衡→建议 |
| 学术研究 | IMRAD + [Stanford 白皮书指南](https://law.stanford.edu/wp-content/uploads/2015/04/White-Papers-Guidelines.pdf) | 引言→文献→方法→发现→讨论→结论 |
| 商业分析 | Porter 五力 + SWOT + 商业画布 | 市场→竞争→模式→财务→战略 |
| 可行性研究 | [FAO 政策简报指南](https://www.fao.org/4/i2195e/i2195e03.pdf) + 决策树 | 技术→经济→法规→风险→建议 |

### 2.2 模板注入机制

```
report_types/political_commentary.md
  └── ## 大纲约束
      基于 RAND 政策分析框架:
      1. 问题界定与背景（为什么关注）
      2. 关键事件与时间线（发生了什么）
      3. 利益相关方分析（各方立场、动机、能力）
      4. 政策因果链（政策如何传导、产生什么效果）
      5. 争议与替代解释（不同视角、不确定性）
      6. 前景评估与政策含义（对决策者意味什么）

      每章必须包含:
      - 至少一个 Mermaid 图表
      - 利益相关方立场对照表
      - 争议/不确定性标注
```

### 2.3 STORM 式多视角大纲生成

借鉴 [Stanford STORM](https://github.com/stanford-oval/storm) 的 perspective-guided 方法改造 Step2：

```
当前: LLM 直接生成大纲
改造:
  1. 识别 3-5 个视角（政策制定者、批评学者、地区专家、受影响群体、国际观察者）
  2. 模拟各视角提问："作为 X，我最想了解什么？"
  3. 用问题列表指导大纲生成，确保覆盖多元视角
  4. 大纲中标注每节回应哪个视角的关切
```

## 三、评估层改造

### 3.1 Rubric-Driven 评估

**参考**：
- [Rubric Is All You Need (ACM 2025)](https://arxiv.org/html/2503.23989v1) — 领域专属 rubric 比通用标准效果好
- [REVISEVAL (ICLR 2025)](https://openreview.net/pdf?id=1tBvzOYTLF) — 修正式参考评估
- [G-Eval](https://www.confident-ai.com/blog/g-eval-the-definitive-guide) — Chain-of-Thought 评估提升相关性

**改造 Step3b**：

```
当前: 1 次 LLM 调用，5 维度同时评
改造:
  Phase 1: 加载领域 rubric (evaluation_rubric.md)
  Phase 2: 5 维度并行评估（每维度独立调用 reasoning 模型）
    - text_quality: 专注语言、术语、AI 味检测
    - corpus_coverage: 专注语料匹配、幻觉检测
    - internal_logic: 专注因果推理、论证链
    - chapter_progression: 专注结构、递进、衔接
    - focus_emphasis: 专注重点突出、信噪比
  Phase 3: Chain-of-Verification 事实验证（Perplexity 搜索关键断言）
  Phase 4: 汇总评分 + 章节级问题定位
```

**每维度 rubric 示例（政治评论 - internal_logic）**：

```markdown
## 内在逻辑评分标准

### 90-100（优秀）
- 每个因果推断都有明确的传导机制说明
- 对混淆变量有讨论
- 区分相关性与因果性
- 反事实分析

### 70-89（良好）
- 主要论点有证据支撑
- 因果链基本完整
- 对立论据有提及

### < 60（不合格）
- 出现"因为 A 所以 B"但未解释传导机制
- 将时间先后等同于因果
- 忽略明显的替代解释
```

### 3.2 Chain-of-Verification

在评估阶段增加事实验证：

```
1. 从报告中提取所有事实断言（数据、日期、人名、事件）
2. 对每个断言生成验证查询
3. Perplexity 搜索验证
4. 标注: ✅ 已验证 / ⚠️ 未找到来源 / ❌ 与搜索结果矛盾
5. 将验证结果注入 Step5 改写 prompt
```

### 3.3 G-Eval 评分校准

参考 G-Eval 的 Chain-of-Thought 方法提升评分一致性：

```
当前: "请打分 0-100"
改造: "请按以下步骤评估:
  1. 列出本章的主要论点（3-5 个）
  2. 对每个论点检查: 有无证据支撑？逻辑是否自洽？
  3. 列出发现的问题
  4. 基于问题数量和严重程度，给出分数
  5. 输出: {reasoning: '...', score: N, issues: [...]}"
```

## 四、呈现层改造

### 4.1 Mermaid 图表规范

**在 Step5/Step7 prompt 中注入**：

```markdown
# 图表生成规范

在以下场景必须插入 Mermaid 图表：

1. **时间线**：涉及 3+ 个有时间顺序的事件
   ```mermaid
   timeline
       title 事件名称
       YYYY-MM-DD : 事件1
       YYYY-MM-DD : 事件2
   ```

2. **关系图**：涉及 3+ 个利益相关方
   ```mermaid
   graph TD
       A[方A] -->|关系| B[方B]
   ```

3. **流程图**：涉及因果链或政策传导
   ```mermaid
   flowchart LR
       A[原因] --> B[传导] --> C[结果]
   ```

4. **对比**：涉及多选项权衡
   用 Markdown 表格，不用图表

每章至少 1 个图表或结构化表格。
```

**DOCX 渲染方案**：
- 方案 A：`mermaid-cli` 本地渲染为 PNG → 嵌入 DOCX（需 Node.js）
- 方案 B：Mermaid.ink 在线 API 渲染 → 下载 PNG 嵌入（无需安装）
- 方案 C：HTML 导出时原生支持，DOCX 只保留文本描述

**参考**：
- [Mermaid.js](https://github.com/mermaid-js/mermaid)
- [LLMermaid](https://github.com/fladdict/llmermaid) — 用 Mermaid 流程图控制 LLM 行为
- [GenAIScript Mermaid 修复](https://microsoft.github.io/genaiscript/blog/mermaids/) — 自动修复 LLM 生成的 Mermaid 语法错误

### 4.2 表格规范

```markdown
# 表格使用规范

以下内容必须用表格呈现：
- 数据对比（不用列表）
- 利益相关方立场（方 | 立场 | 动机 | 行动 | 来源）
- 时间线事件（日期 | 事件 | 来源 | 影响）
- 政策选项对比（选项 | 优势 | 劣势 | 可行性）

禁止：
- 将可以表格化的内容用 bullet list 呈现
- 表格超过 8 列（影响可读性）
```

### 4.3 语言规范

```markdown
# 中英混合处理规范

## 专有名词
- 首次：中文译名（英文原名），如"贾利斯科新世代卡特尔（CJNG）"
- 后续：缩写或中文简称

## 引述
- 保留英文原文 + 中文翻译
- 如：美方声明称"non-state armed groups"（非国家武装团体）

## 禁止
- 整段英文不翻译
- 中英混用半句话（如"进行了counter cartel行动"）
- ISO 时间戳、裸 URL 出现在正文
```

## 五、Prompt 工程持续优化

### 5.1 结构化 Prompt 模板

参考 [IBM Prompt 指南](https://www.ibm.com/think/prompt-engineering) 和 [Lakera 指南](https://www.lakera.ai/blog/prompt-engineering-guide)：

```
当前: 每个 Step 的 prompt 硬编码在 Python 文件中
改造:
  prompts/
  ├── step2_outline.md          ← 大纲生成 prompt 模板
  ├── step3b_eval_{dimension}.md ← 各维度评估 prompt
  ├── step5_rewrite.md          ← 改写 prompt 模板
  ├── step7_style_{type}.md     ← 各类型风格化 prompt
  └── common/
      ├── mermaid_rules.md      ← 图表生成规范
      ├── table_rules.md        ← 表格规范
      └── language_rules.md     ← 语言规范
```

**优势**：
- Prompt 可独立迭代，不改 Python 代码
- 支持 A/B 测试不同 prompt 版本
- 用户可自定义 prompt 模板

### 5.2 Prompt 版本管理与评测

```
prompts/
├── v1/                  ← 当前版本
├── v2/                  ← 实验版本
└── eval_results/
    ├── v1_kateer.json   ← v1 prompt 在 kateer 上的评分
    └── v2_kateer.json   ← v2 prompt 对比
```

## 六、质量迭代闭环

### 6.1 完整闭环流程

```
v1 → Step3b(评估, rubric-driven)
        ↓
    总分 ≥ 70 且各维度 ≥ 60?
        ↓ 否                    ↓ 是
    Step5(改写, 注入评估)     直接 Step6
        ↓
    Step3b(再评估, 只评低分维度)
        ↓
    达标或到上限(2轮)?
        ↓ 是
    Step6 → Step7 → Step9
```

### 6.2 成本控制

| 轮次 | 改写范围 | 评估方式 | 额外成本 |
|---|---|---|---|
| 第 1 轮 | 全部章节 | 5 维度全评 + CoV | 基准 |
| 第 2 轮 | 仅 < 60 分章节 | 只评低分维度 | +15-30% |
| 合计 | | | 基准 × 1.3 |

### 6.3 退出条件

- 总分 ≥ 70 且各维度 ≥ 60 → 进入 Step6
- 迭代 2 轮未达标 → 输出当前最佳版本 + 警告
- 第 2 轮分数未提升 → 停止迭代，避免无效消耗
